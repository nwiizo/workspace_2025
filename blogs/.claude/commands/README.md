# Custom Commands for Blog Project

このディレクトリには、**深く考え、より良いブログを書く**ためのカスタムコマンドが含まれています。

## 哲学：深く考えることから始める

良いブログは「書く」前に「考える」ことから始まります。このコマンド体制は、以下の3つのフェーズで構成されています：

```
【Phase 1】深く考える（Think）
 ↓
【Phase 2】構造化して書く（Write）
 ↓
【Phase 3】多角的にレビューする（Review）
```

---

## Phase 1: 深く考える（Think）

**書く前に、徹底的に思考を深める**

### `/deep-thinking-prompt` - 深い思考のための問いかけ
**ファシリテーター**: 田中慎也（哲学者・43歳）

**7つの問いかけ**:
1. 原体験への問いかけ - なぜこのテーマに興味を持ったのか
2. 前提への問いかけ - 当たり前だと思っていることは何か
3. 対立への問いかけ - 矛盾や葛藤はどこにあるか
4. 構造への問いかけ - システムとしてどう機能しているか
5. 時間軸への問いかけ - 過去・現在・未来でどう変化しているか
6. 他者への問いかけ - 異なる立場の人はどう見ているか
7. 本質への問いかけ - 結局、何が問題なのか

**使い方**:
```
/deep-thinking-prompt "読書について書きたい"
```

**こんな時に**:
- 何について書くか決まったが、何を伝えたいかまだ明確でない時
- 表面的な理解から、本質的な洞察に深めたい時
- 自分だけの視点を見つけたい時

---

### `/critical-perspective` - 批評的視点の導入
**ファシリテーター**: 村上聡（社会学者・47歳）

**4つの批評的視点**:
1. 「べき論」の解体 - 表面的な規範を疑う
2. 構造的問題の認識 - システムレベルでの問題を見抜く
3. 純粋な動機への回帰 - 本来の目的と外的圧力を区別する
4. 多様性と自由の尊重 - 一つの正解を押し付けない

**使い方**:
```
/critical-perspective "読書について"
/critical-perspective draft.md
```

**こんな時に**:
- 「〜すべき」という押し付けに疑問を持った時
- 個人の問題を、システムの問題として読み替えたい時
- 多様性と自由を擁護する記事を書きたい時
- 批評的・社会学的な視点を加えたい時

---

## Phase 2: 構造化して書く（Write）

**思考を整理し、読者に届く形にデザインする**

### `/structural-thinking` - 構造設計支援
**アーキテクト**: 佐藤美咲（情報アーキテクト・40歳）

**5つのフェーズ**:
1. 核心の抽出 - 一言で何を伝えるか
2. 読者理解マップ - 誰に、何を届けるか
3. 情報のグルーピング - 何を、どのまとまりで
4. 構造パターンの選択 - 問題解決型/物語型/対比型/探求型
5. 流れの設計 - 読者をどう導くか

**使い方**:
```
/structural-thinking "読書についての考えをまとめたい"
/structural-thinking draft.md
```

**こんな時に**:
- 書きたいことはあるが、どう構成すべきか分からない時
- 既存のdraftを整理し直したい時
- 論理的で読みやすい流れを作りたい時

---

### `/title-workshop` - タイトル案生成
**クリエイター**: 高橋瞳（コピーライター・35歳）

**5つのアプローチ**:
1. 問いかけ型 - なぜ/どうすれば/本当に
2. 対比・逆説型 - 〜ではなく/〜のに
3. 数字・具体型 - 3つの方法/10年で分かった
4. 共感・体験型 - 〜に疲れたあなたへ
5. 宣言・断定型 - 〜すべきだ/〜が全てを変える

**使い方**:
```
/title-workshop blog.md
/title-workshop "読書についての記事"
```

**こんな時に**:
- 記事は完成したが、タイトルが決まらない時
- 複数のタイトル案から選びたい時
- クリック率を上げたい時

---

### `/textlint-polish` - 文章校正・AIっぽさ除去
**エディター**: 岡田翔太（文章校正スペシャリスト・35歳）

**自動チェック項目**:
1. AIっぽい表現の検出・除去
2. 比喩的・詩的すぎる表現の簡潔化
3. 冗長な言い回しの削減
4. 文体の統一（です・ます調）
5. 弱い表現の検出

**使い方**:
```
/textlint-polish blog.md
```

**こんな時に**:
- **初稿完成後、レビュー前に必ず実行**
- AIっぽい表現が混ざっていないか確認したい時
- 文章の基本的な品質を自動チェックしたい時
- 冗長性を機械的に削減したい時

**Textlint実行コマンド**:
```bash
# エラーチェックのみ
npx textlint blog.md

# 自動修正を適用
npx textlint --fix blog.md
```

## 🚀 全自動レビュー（推奨）

### `/full-review` - すべてのレビューを自動実行
**統括マネージャー**: 鈴木麻里（編集長・42歳）

**実行フロー**（自動→承認→修正の3段階）:

**Phase 1: 自動レビュー（8-15分）**
1. Textlint診断（診断のみ、`--fix`なし）
2. 初学者フィードバック（Tier 1）
3. 総合品質レビュー（Tier 1）
4. 記事タイプ別レビュー（Tier 2）
5. 統合改善提案レポート生成

**Phase 2: ユーザー承認（ユーザー操作）** ⭐重要
```
📋 レビューが完了しました。

🔴 最優先改善: XX件
🟡 重要改善: XX件
🟢 任意改善: XX件

【ユーザーの選択肢】
1️⃣ 最優先のみ自動修正する（推奨）
2️⃣ 最優先+重要を自動修正する
3️⃣ すべて自動修正する
4️⃣ 詳細レポートを見てから決める
5️⃣ 手動で修正する（自動修正しない）

どれを選択しますか？（1-5）
```

**Phase 3: 承認された修正の適用（1-10分）**
- ユーザーが選択した範囲の修正のみ適用
- 差分確認と最終確認

**使い方**:
```
/full-review blog.md コラム
/full-review draft.md 技術
/full-review article.md
```

**こんな時に**:
- **最も推奨**: 初稿完成後、一括でレビューしたい時
- 複数のコマンドを手動で実行するのが面倒な時
- 包括的な改善提案が欲しい時
- 公開前の最終チェックに

**実行時間**: 約15-40分（レビュー8-15分 + 承認5-10分 + 修正1-10分）

## Phase 3: 多角的にレビューする（Review）

**6人の専門家 + 1人の初学者が、異なる視点で記事をレビュー**
**※ 個別実行も可能ですが、`/full-review`での一括実行を推奨**

### ⭐ 優先度について

レビューコマンドには優先度があります：

- **Tier 1（必須）**: すべての記事で必ず実行すべき
- **Tier 2（推奨）**: 品質を高めるために推奨
- **Tier 3（オプション）**: 特定の目的がある場合のみ

---

### Tier 1（必須）: `/beginner-feedback` ⭐⭐⭐
**読者代表**: 佐々木ゆい（28歳・一般読者）

**最優先の理由**:
- 専門家の目では見えない「伝わらない」を発見
- 一般読者の素朴な疑問を明らかにする
- 共感できるか、分かりやすいかを確認
- **どんな記事でも、最初にこれを実行すべき**

**4つの評価軸**（各0-10点）:
1. 第一印象 - パッと見て読みたいと思うか
2. 分かりやすさ - 普通に読んで理解できるか
3. 共感度 - 自分のことのように感じられるか
4. 読後感 - 読み終えてどう感じたか

**使い方**:
```
/beginner-feedback blog.md
```

**こんな時に**:
- **必ず最初に実行**（一般読者に届くかを確認）
- 専門用語が多いと感じる時
- 論理展開に自信がない時
- 初稿を書き上げた時

---

### Tier 1（必須）: `/blog-quality-review` ⭐⭐⭐
**レビュアー**: 山本理恵（テクニカルライター・38歳）

**6つの評価軸**（各0.0-5.0点）:
1. 防御力 - 批判や反論への耐性
2. 思考整理力 - 情報の論理的構造化
3. 実践応用性 - 読者が行動に移せる価値
4. 構成と読みやすさ - 視覚的要素と文体
5. コミュニケーション力 - 人間味のある伝達
6. 人間らしさ - 温度感と個性

**使い方**:
```
/blog-quality-review blog.md
```

**こんな時に**:
- **必ず実行**（総合的な品質確認）
- 初稿を書き上げた時
- 全体的な品質を客観的に評価したい時
- 具体的な改善ポイントを知りたい時

---

### Tier 2（推奨）: `/redundancy-check` ⭐⭐
**レビュアー**: 中村鋭一（元新聞記者・45歳）

**推奨の理由**:
- 冗長性は読者の時間を奪う
- 情報密度を高めることで読みやすさ向上
- 公開前の最終チェックに最適

**4つの評価軸**（各0.0-5.0点）:
1. 情報密度 - 1文あたりの情報量
2. 簡潔性 - 冗長表現・無駄な修飾の少なさ
3. 論理効率 - 論理的重複・循環論法の少なさ
4. 構造最適性 - セクション構成の必要十分性

**使い方**:
```
/redundancy-check blog.md
```

**こんな時に**:
- 文章が長すぎると感じる時
- 同じことを繰り返している気がする時
- **公開前の最終簡潔化に必須**

---

### Tier 2（推奨）: `/narrative-empathy-review` ⭐⭐
**レビュアー**: 森川由紀（小説家・44歳）

**6つの評価観点**:
1. 物語構造 - 起承転結が機能しているか
2. エモーショナル・ジャーニー - 読者の感情の旅路
3. 共感ポイント - 「自分のことだ」と感じる箇所
4. 読者変容 - 読む前と後での変化
5. 感情的解像度 - 読後に残る感情の質と深さ
6. 読後感 - 記事を読み終えた瞬間の余韻

**使い方**:
```
/narrative-empathy-review blog.md
```

**こんな時に**:
- 読者の心を動かしたい時
- 物語性を強化したい時
- 読後感を深めたい時

---

### Tier 2（推奨）: `/punchline-review` ⭐⭐
**レビュアー**: 小林剛（コピーライター・52歳）

**推奨の理由**:
- 記憶に残るフレーズは拡散力を高める
- キーメッセージの明確化で理解度向上
- SNSシェア時の引用しやすさが重要

**5つの評価観点**:
1. 粘着性 - フレーズが記憶に残る度合い
2. 引用可能性 - ビジネス・日常で引用されやすさ
3. SNSシェア適性 - SNSで拡散されやすさ
4. 核心メッセージの明確さ - 各章のキーメッセージ
5. 具体性と抽象性のバランス - 概念の端的な表現

**使い方**:
```
/punchline-review blog.md
```

**こんな時に**:
- 記憶に残るフレーズを強化したい時
- SNSでシェアされやすい表現を作りたい時
- キーメッセージを明確にしたい時

---

### Tier 3（オプション）: `/ai-humanity-check` ⭐
**専門家**: AI検出専門家

**オプションの理由**:
- 特定の懸念がある場合のみ必要
- 人間らしさは他のレビューでもチェック可能
- 機械的な表現が気になる場合に実施

**AIっぽさスコア**（0.0-5.0、低いほど人間らしい）:
- 文体の自然さ
- 感情表現の真実味
- 具体的な失敗談や体験
- 個性とオリジナリティ

**使い方**:
```
/ai-humanity-check blog.md
```

**こんな時に**:
- AI生成疑惑を避けたい時
- 人間味のある文章にしたい時
- 機械的な表現が多いと感じる時

## 推奨ワークフロー

### 🌟 完全版フロー（じっくり書く場合）

```
【Phase 1: 深く考える】
1. /deep-thinking-prompt "テーマ"
   ↓ 数日考える、対話する、メモする

2. /critical-perspective "テーマ"（コラム記事で「べき論」を扱う場合）
   ↓ 批評的視点を加える

3. /structural-thinking "テーマ"
   ↓ 構造設計に基づいて執筆開始

【Phase 2: 書く】
4. 初稿を書く
   ↓

5. /textlint-polish draft.md
   ↓ AIっぽい表現・冗長性を自動修正

6. /title-workshop draft.md
   ↓ タイトル候補から選ぶ

【Phase 3: レビュー（Tier 1 - 必須）】
7. /beginner-feedback draft.md ⭐⭐⭐
   ↓ 一般読者視点で分かりやすさ確認

8. /blog-quality-review draft.md ⭐⭐⭐
   ↓ 総合品質チェック、改善点を修正

【Phase 4: レビュー（Tier 2 - 推奨）】
9. 記事タイプに応じて選択:
   - コラム: /narrative-empathy-review draft.md
   - 技術: /redundancy-check draft.md
   ↓ さらに品質向上

10. /punchline-review draft.md（必要に応じて）
    ↓ キーメッセージを磨く

【Phase 5: 最終チェック（Tier 3 - オプション）】
11. /ai-humanity-check draft.md（気になる場合のみ）
    ↓ 最終調整

12. 公開！
```
