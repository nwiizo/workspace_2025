# Structural Thinking - 構造的思考支援

このコマンドは、ブログ記事の構造を設計するための思考支援ツールです。「何をどの順番で書くか」を明確にし、論理的で読みやすい構成を作ります。

## 使用方法

```
/structural-thinking [テーマまたはファイルパス]
```

例:
```
/structural-thinking "読書について書きたい"
/structural-thinking draft.md
```

## このコマンドの目的

**思考を構造化し、読者に届く形にデザインする**

- 散らばった思考を整理する
- 論理的な流れを作る
- 読者の理解プロセスに合わせる
- 「伝わる」構成を設計する

## 構造設計の5つのフェーズ

### Phase 1: 核心の抽出

**この記事で伝えたいことは、一言で何か？**

- コアメッセージを1文で
- 読者の Before → After を明確に
- なぜ「今」この記事が必要か

### Phase 2: 読者理解マップ

**読者は誰で、何を求めているか？**

```
読者のペルソナ:
- 年齢層: [...]
- 抱えている問題: [...]
- 知識レベル: [...]
- 期待すること: [...]

読者の旅路:
記事を開く前 → [感情状態]
読んでいる最中 → [感情の変化]
読み終えた後 → [期待される変化]
```

### Phase 3: 情報のグルーピング

**何を、どのまとまりで伝えるか？**

```
必須要素:
- [必ず伝えるべきこと1]
- [必ず伝えるべきこと2]
- [必ず伝えるべきこと3]

補助要素:
- [あると良い情報1]
- [あると良い情報2]

削除候補:
- [なくても良い情報1]
- [なくても良い情報2]
```

### Phase 4: 構造パターンの選択

**どんな構造が最適か？**

#### パターン1: 問題解決型
```
1. 問題提起（読者の痛み）
2. 問題の深掘り（なぜこの問題が起きるか）
3. 解決策の提示（具体的な方法）
4. 実践へのステップ（次にすべきこと）
```

#### パターン2: 物語型
```
1. 導入（共感できるエピソード）
2. 展開（問題の発見と葛藤）
3. 転換（気づきと変化）
4. 結末（学びと次への一歩）
```

#### パターン3: 対比型
```
1. 一般的な理解（多くの人が思っていること）
2. 実は違う（前提を疑う）
3. 本質は何か（新しい視点）
4. どうすべきか（実践的示唆）
```

#### パターン4: 探求型
```
1. 問いの提示（なぜ？どうして？）
2. 仮説1の検証（こう考えられるが...）
3. 仮説2の検証（別の視点では...）
4. 統合（結局、何が言えるか）
```

### Phase 5: 流れの設計

**読者をどう導くか？**

```
導入（Hook）:
- [読者の注意を引く問い/エピソード]
- [この記事を読む価値の提示]

本論（Body）:
セクション1: [...]
├─ 目的: [このセクションで何を伝えるか]
├─ 方法: [どう伝えるか（論理/体験/データ）]
└─ 移行: [次のセクションへの橋渡し]

セクション2: [...]
├─ 目的: [...]
├─ 方法: [...]
└─ 移行: [...]

セクション3: [...]
├─ 目的: [...]
├─ 方法: [...]
└─ 移行: [...]

結論（Closure）:
- [要約（繰り返しではなく、昇華）]
- [読者への問いかけ/行動喚起]
```

## ワークフロー

### Step 1: 現状分析

**既存のdraft.mdがある場合**:
- 現在の構造を可視化
- 情報の重複を発見
- 論理の飛躍を特定
- 不足している要素を洗い出し

**テーマだけの場合**:
- ユーザーの意図を確認
- 伝えたいことをリストアップ
- 読者像を明確化

### Step 2: 構造設計

**最適な構造パターンを提案**

```
推奨構造: [問題解決型/物語型/対比型/探求型]
理由: [なぜこの構造が最適か]
```

**セクションごとの設計**

```
## セクション1: [タイトル]
目的: [何を伝えるか]
内容: [どう伝えるか]
読者の状態: [このセクション後、読者はどう変わるか]
文字数目安: XXX字

## セクション2: [タイトル]
...
```

### Step 3: チェックリスト

**構造の品質を確認**

```
□ コアメッセージは明確か
□ 導入部で読者を引き込めるか
□ 論理の飛躍はないか
□ 情報の重複はないか
□ 読者の理解プロセスに沿っているか
□ 各セクションの目的は明確か
□ 結論は読後感を残すか
□ 全体で2000-3000字程度か（長すぎないか）
```

## 使用上の注意

1. **柔軟に変える**: 設計通りに書く必要はない。書きながら調整してOK
2. **読者中心**: 自分の都合ではなく、読者の理解プロセスを優先
3. **シンプルに**: 複雑な構造は避ける。見出しは2階層まで
4. **文字数を意識**: 長すぎる記事は読まれない。2000-3000字が目安
